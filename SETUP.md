# Setup / 実運用開始手順

## 1. この手順の目的

部門別損益ダッシュボードを、WFSの保存・開発ルールに従って安全に実運用へ移す手順です。

```text
SSDで設計・開発・テスト
        ↓
GitHubへPush（設計・履歴の正式版）
        ↓
Google DriveのTemplatesを更新（親テンプレート）
        ↓
Customersへコピー（実運用ファイル）
```

GitHubからのcloneと、Googleスプレッドシートの導入は別の作業です。利用者が日常業務だけを行う場合、GitHubのcloneは不要です。

## 2. 保存場所と役割

| 保存場所 | 役割 | 保存するもの |
|---|---|---|
| SSD | 開発工房 | cloneしたリポジトリ、設計変更、テスト |
| GitHub | 設計図書館 | 設計書、README_AI、テスト、変更履歴、コード |
| Google Drive / Templates | 親テンプレート | `TPL_部門別損益ダッシュボード_v0.5` |
| Google Drive / Customers | 実運用 | 会社・案件ごとにコピーしたスプレッドシート |
| 各会社統合DB | 業務データの正本 | 将来連携する会計・業務データ |

## 3. SSDへcloneする（開発・保守担当者のみ）

1. SSDに作業フォルダ `D:\WellzFriend_projects` があることを確認します。
2. ターミナルで次を実行します。

```powershell
cd D:\WellzFriend_projects
git clone https://github.com/wellzfriend420/wellzfriend-department-pl-dashboard.git
cd wellzfriend-department-pl-dashboard
```

3. `README.md`、`README_AI.md`、本ファイル、`tests/TEST_CASES.md`を確認します。

本リポジトリには実行プログラムやインストーラーがないため、パッケージのインストール、ビルド、Apps Scriptのデプロイは不要です。clone後は、設計・検証・変更履歴を保守する作業環境として使用します。

## 4. 親テンプレートから実運用ファイルを作る

1. Google Driveで `WellzFriend/Templates/Department PL Dashboard` を開きます。
2. `TPL_部門別損益ダッシュボード_v0.5` を開き、版が正しいことを確認します。
3. 親テンプレートを直接編集せず、「コピーを作成」を選びます。
4. コピー先を `WellzFriend/Customers/<会社名または案件名>` にします。フォルダがなければ作成します。
5. ファイル名を `<会社名または案件名>_部門別損益ダッシュボード` に変更します。

親テンプレートは原本です。日常入力や顧客固有設定には使用しません。

## 5. SETTINGSを設定する

コピーした実運用ファイルで、次の順に設定します。

1. 会社名
2. 会計年度開始月
3. 表示月
4. データ最終更新日
5. 部門コードと部門名
6. 勘定科目とPL区分の対応表

製造原価のPL区分は `材料費 / 労務費 / 製造経費` です。弥生会計の勘定科目名はSETTINGSへ完全一致で登録します。科目を追加・変更するときは、CALCへ直接式を追加せずマスタを更新します。

## 6. DATAへ明細を入れる

1. `DATA` シートのサンプル入力部分 `A:E` を消去します。
2. 自動計算列 `F:G` は消去しません。
3. `計上日 / 部門 / 勘定科目 / 金額 / 摘要` の順で明細を貼り付けます。
4. 実在する部門名と勘定科目名がSETTINGSと一致していることを確認します。
5. 顧客名、個人情報、認証情報、DB接続情報は入力しません。

サンプル明細と弥生取込明細を同時に残さないでください。同じ年月・科目・部門の重複取込は弥生会計との差異原因になります。

## 7. 運用開始前に確認する

- DATAのPL区分に `未分類` が残っていない。
- CALCの部門合計と全体合計が一致する。
- DASHBOARDへ部門別の当期累計・前年同期累計・利益率が表示される。
- REPORTに部門別累計の材料費・労務費・製造経費が売上直下へ表示される。
- REPORTの製造経費内訳を展開できる。
- REPORTの当期累計が会計年度開始月から表示月までの合計になっている。
- REPORTの前年同期累計が前年の同じ期間になっている。
- 前年同期累計が当期累計の12か月前の同期間を参照している。
- `#REF!`、`#DIV/0!`、`#VALUE!`、`#N/A`などのエラーがない。

確認内容は [tests/TEST_CASES.md](tests/TEST_CASES.md) に沿って実施します。合格後、実運用を開始します。

## 8. 改修・バージョンアップ

1. SSDのcloneで設計資料・仕様・テストを更新します。
2. テスト合格後、README_AIとCHANGELOGを更新します。
3. GitHubへコミット・Pushします。
4. Google Driveの親テンプレートを更新し、ファイル名のVersionを上げます。
5. HubのBusiness Template RegistryにVersionと確認日を記録します。
6. 新規案件は更新済み親テンプレートからコピーします。

既存のCustomersファイルは自動更新されません。運用データを保護するため、変更点を確認してバックアップを作成し、案件ごとに移行します。

## 9. 禁止事項

- 親テンプレートを顧客運用に直接使用しない。
- Customersの運用ファイルをGitHubへ保存しない。
- 顧客・個人・契約・請求・認証情報をリポジトリへコミットしない。
- CALC、DASHBOARD、REPORTの数式を利用者判断で直接変更しない。
- 廃止したテンプレートVersionを削除・再利用しない。
